# 对话质量调优与目标判定 —— 项目核心难点

> 本文档是整个培训中台项目最有技术深度的部分。结构框架做得再好，面试官追问的最终都会落到"AI 对话到底怎么做好的"。

---

## 一、对话质量调优：不是写个 Prompt 就完事

### 1.1 问题本质

AI 陪练对话面临的核心矛盾：**角色扮演的"演技"与培训效果之间的平衡**。

| 如果 AI 太强（演技太好） | 如果 AI 太弱（配合太多） |
|-------------------------|------------------------|
| 学员被虐得说不出话，没有训练效果 | 学员轻松通关，没有被挑战，能力不提升 |
| 回复太长、太复杂，学员接不住 | 回复太简单，不像真实客户 |
| 一直制造新阻力，对话永远推不下去 | 几句话就妥协，学员学不到处理异议的能力 |

### 1.2 我们踩过的坑及对应的解决方案

#### 坑 1：AI 无限拖延确认身份

**现象**：学员说"您好我是XX公司的"，AI 反复问"你是谁？你怎么有我电话？你先说清楚"，导致前 10 轮都在确认身份，根本进入不了业务环节。

**根因**：Prompt 里写了"保持防备"，LLM 过度解读为"永远不配合"。

**解决方案**（体现在 Prompt 第 7 条规则）：
```
配合节奏：对方已说明机构/岗位（或工号）且说明来电与账户相关，
并问是否本人等确认身份的问题时——须正面配合（是/否或配合核对一项信息），
禁止再用「太含糊」「套信息」「说不清楚」笼统堵回；
整段确认身份内最多1句短反问，下一轮必须配合。
```

**设计原则**：**明确量化约束**。"保持防备"是模糊指令，"最多1句反问，下一轮必须配合"是可执行的硬规则。LLM 在遵循硬规则上远比遵循模糊描述靠谱。

---

#### 坑 2：AI 回复太长，像在念课文

**现象**：AI 一次回复 200+ 字，条理清晰地解释自己的困难和诉求。这不像真实客户（真实客户是情绪化的短句）。

**根因**：LLM 的默认行为是"尽量完整地回答"，与真实对话场景相反。

**解决方案**（多层约束叠加）：

1. **硬性字数限制**：
```
控制回复长度：每次回复控制在 20-40 字，短句为主
```

2. **口语化词库引导**（在 PROCESS 模板中）：
```
借款人常用表达（从这些词里选，不要用书面语）：
- 没钱时：手头紧、凑不出来、拿不出、没那个钱
- 想还时：我不是赖账的人、钱肯定会给的
- 嫌急时：太急了、来不及、给我点时间
- 不满时：天天打电话、烦死了
- 求通融：跟你们那边说说、通融一下
- 质疑时：利息怎么这么高、这钱怎么算的
```

3. **输出前自检**（Prompt 末尾）：
```
输出前必检：
1. 有没有催收行业术语？→ 换成借款人的日常口语
2. 是否正面回应了对方最新一句？→ 先答再补态度
3. 有没有说出角色不该知道的信息？→ 只说借款人知道的事
```

4. **代码层后处理**（`sanitizeAiReply`）：
```java
public String sanitizeAiReply(String raw) {
    // 去掉角色前缀：如 "[客户]：" "[AI]：" 等
    // 去掉 markdown 格式
    // 去掉引号包裹
    // 截断过长回复（兜底）
}
```

**设计原则**：**不依赖单一手段**。字数约束 + 词库引导 + 自检指令 + 代码后处理，四层防御。任何一层失效，其他层仍在生效。

---

#### 坑 3：AI 用"反问"代替"回答"

**现象**：学员问"你方便什么时候还款？"，AI 回"你们怎么知道我逾期了？你先说清楚"——明明应该先回答问题再表达态度，结果变成了用反问来回避。

**根因**：LLM 在角色扮演中倾向于用"质疑"来体现"防备"，因为质疑是最容易产生的对抗行为。

**解决方案**（Prompt 第 3 条规则 + 开头强化指令）：
```
# 开头第一段（最高权重位置）
每轮优先读懂对方最新一句：他问什么、要你确认什么，就先正面回答
（是/否、能/不能、配合核对等），再用短句补态度；
减少无端质疑与连环反问，不要用反问代替回答。

# 规则第 3 条
回应优先、少质疑：以回答对方当前问题或陈述为主，
用陈述、表态、简短补充推进；
不要用连环反问、质疑身份与来意来拖节奏。
```

**设计原则**：**关键行为约束放在 Prompt 的最高权重位置**（开头第一段）。LLM 对 Prompt 开头和结尾的注意力最高，中间段容易被忽略。所以核心行为约束必须出现在开头。

---

#### 坑 4：AI 泄露"角色不该知道的信息"

**现象**：AI 扮演借款人，但回复中出现了"还款意愿"、"减免方案"、"核身"、"委外"等催收行业术语。真实客户不可能知道这些词。

**根因**：LLM 的训练数据中催收相关内容会包含行业术语，角色扮演时容易"穿帮"。

**解决方案**（Prompt 第 6 条 + 显式禁用词列表）：
```
角色认知边界：你是普通借款人，不了解催收行业内部流程和专业术语。
以下词汇禁止出现在你的回复中：
还款意愿、还款能力、还款方案、减免方案、协商方案、结清方案、
核身、配合核身、配合度、逾期天数、委外、委案、记录困难、
报备、工单、缓冲期、分期方案、一次性结清。
用日常口语替代（如"我不是不想还""能不能少还点""5到7天太急了"）。
```

**设计原则**：**显式枚举禁止项 + 提供替代方案**。光说"不要用专业术语"不够具体，LLM 不知道哪些算专业术语。必须列出禁用词表，并给出对应的口语替代，让 LLM 有路可走。

---

#### 坑 5：情绪变化不自然（要么一直对抗，要么突然妥协）

**现象**：AI 前面一直态度强硬，突然某一轮就完全配合了，没有渐变过程。或者反过来，学员已经很专业了 AI 还在无脑对抗。

**解决方案**（情绪递进规则 —— 5 阶段模型）：
```
情绪递进规则：
1. 初始防备期（开始）：冷淡、警觉、短句回应
2. 试探松动期（学员展现专业度后）：防备降低，开始主动说几句
3. 方案纠结期（进入还钱讨论）：在"想解决"和"舍不得钱"之间反复纠结
4. 异议波动期（触碰敏感点时）：情绪突然上升
5. 理性回归期（异议被化解后）：重新回到理性

如果学员的沟通技巧差，你的情绪可以在第1-2阶段反复停滞，
甚至直接提出要挂电话。情绪进展取决于学员的表现，不要自动推进。
```

**设计原则**：**情绪是对学员表现的响应，不是自动推进的脚本**。最后一句话是关键——告诉 LLM "不要自动推进"，否则它倾向于按 1→2→3→4→5 顺序走完。

---

#### 坑 6：RAG 知识"污染"角色扮演

**现象**：RAG 检索返回了催收话术（"您好，我是XX公司..."），AI 的回复中突然冒出了催收员的口吻（因为 RAG 内容进入了 Prompt）。

**解决方案**（RAG 使用说明 + 角色隔离）：
```
## RAG 知识参考（可选）
以下内容由知识库检索，使用催收培训的书面表述。
你说话时必须转换为借款人的日常口语，
不得直接复用其中的专业词汇。若本节为空则忽略。

## 标准话术参考
以下是催收员视角的标准话术参考，
仅供理解意图，禁止复用其措辞：
```

**设计原则**：**告诉 LLM "这段话是别人的视角"**。不能光把 RAG 结果塞进去，必须明确标注"这是催收员视角的参考，你是借款人，要转换立场"。

### 1.3 对话质量的工程化保障

#### sanitizeAiReply（代码层净化）

```java
public String sanitizeAiReply(String rawReply) {
    if (StringUtils.isBlank(rawReply)) return "";
    String cleaned = rawReply;
    // 1. 去掉角色前缀标签
    cleaned = cleaned.replaceAll("^\\[?(客户|AI|学员|Customer|Trainee)\\]?[：:]\\s*", "");
    // 2. 去掉 markdown 格式
    cleaned = cleaned.replaceAll("\\*\\*|__|~~", "");
    // 3. 去掉首尾引号
    cleaned = cleaned.replaceAll("^[\"'「]|[\"'」]$", "");
    // 4. 压缩多余空白
    cleaned = cleaned.replaceAll("\\s+", " ").trim();
    return cleaned;
}
```

#### 字数检测与告警

对话回复如果超过 100 字（超出 20-40 字约束的 2.5 倍），记录 warn 日志并在监控面板中标注，作为 Prompt 退化的信号。

## 二、目标判定：分离式架构设计

### 2.1 为什么目标判定必须独立

**错误设计（我们最初的方案）**：让对话 AI 同时输出回复和判定结果。

```json
// 错误：让一个 LLM 同时做两件事
{
  "reply": "（角色扮演回复）",
  "goalAchieved": true,
  "reason": "学员已确认身份"
}
```

**问题**：
1. **角色混淆**：AI 在扮演客户的同时还要"跳出来"做裁判，两个角色互相干扰
2. **幻觉**：AI 倾向于把对话"演得更好"而过早判定达成（self-serving bias）
3. **输出不稳定**：要求同时输出自然语言回复和结构化判定，格式错误率翻倍

**正确设计（当前方案）**：对话生成和目标判定完全分离为两次独立 LLM 调用。

```
sendMessage() → 对话 LLM（角色扮演，纯文本输出）
    ↓ (返回 AI 回复给前端)
judgeGoal()  → 判定 LLM（裁判角色，JSON 输出）
    ↓ (返回 achieved/failed)
```

### 2.2 目标判定 Prompt 的设计精髓

```
你是目标判定器，只根据规则判断 achieved / failed / sessionShouldEnd。

# 判定流程（固定顺序）
1. 判断结束信号 → sessionShouldEnd
2. 判断是否存在学员严重违规 → failed
3. 若未违规，判断达成条件 → achieved
```

**三个关键设计决策**：

#### 决策 1：固定顺序判定

必须按 结束信号 → 违规 → 达成 的固定顺序。为什么？

- 如果先判断达成再判断结束：学员说"好的再见"可能被判为"确认了某条件"（因为 achieved 的诱惑力更大）
- 如果先判断达成再判断违规：学员说"你TM给我减到本金"可能因为"确实提出了还款意愿"而被判达成

固定顺序确保：有结束信号 → 直接结束（不管是否达成）；有违规 → 直接失败（不管是否部分达成）。

#### 决策 2："未达成 ≠ 失败"的明确区分

```
以下情况不判定失败，返回 achieved=false, failed=false：
- 学员尚未完成目标要求的行为 → 属于"尚未达成"，不等于失败
- 学员发言方向正确但不够完整 → 仍有后续轮次补充机会
```

这是调了很久才加上的约束。早期版本中 LLM 经常把"还没做到"判为"失败"，导致学员刚开始尝试就被判出局。

**设计原则**：**只有严重违规才能触发失败**。"还没做到"只是还没达成——对话应该继续，给学员更多机会。

#### 决策 3：50% 达成阈值

```
所有达成条件逐条核对，如果有多个条件，完成50%及以上就 achieved=true
```

为什么不要求 100%？
- 真实场景中，催收员不可能一次通话完成所有目标
- 如果要求 100%，训练变成了"完美主义考试"，打击学员积极性
- 50% 阈值让学员"做得差不多"就能通过，重点训练核心能力

### 2.3 目标判定的工程化实现

#### 轮次超限保护

```java
// 轮次到限：给 LLM 最后一次达成判定机会，未达成则强制失败
Integer roundLimit = currentGoal.getDialogueRoundsSetting();
if (roundLimit != null && currentGoalRounds >= roundLimit) {
    GoalJudgeResultVO result = callGoalJudgeLlm(...);
    if (result != null && Boolean.TRUE.equals(result.getAchieved())) {
        return result;  // 最后一次判定通过，放行
    }
    // 未通过，强制失败
    GoalJudgeResultVO forcedFailed = new GoalJudgeResultVO();
    forcedFailed.setFailed(true);
    forcedFailed.setSummary("轮次已达上限，目标未达成");
    return forcedFailed;
}
```

**为什么需要轮次保护？**
- 没有轮次限制 → 学员可以和 AI 聊 100 轮也不结束
- 有轮次限制 → 训练有时间压力，更像真实场景
- "最后一次机会"设计 → 避免学员明明最后一句话达成了但因为正好踩线被判失败

#### Bridge 上下文（跨目标衔接）

```java
/**
 * 目标切换后第一轮，当前目标的对话记录可能仅有 AI 过渡回复和学员新消息，
 * 缺少前一目标末尾的上下文。此方法在当前目标学员发言不足 2 条时，
 * 从前一目标末尾提取最后 2 条消息作为衔接上下文。
 */
private String buildJudgeDialogueWithBridge(dialogues, currentIndex, locale) {
    // 当前目标的学员发言不足2条时，补充前一目标的最后2条消息
    if (studentCountInCurrentGoal < 2 && currentIndex > 0) {
        bridge = "[前一目标末尾（仅供理解上下文衔接，不作为当前目标判定依据）]\n"
               + lastTwoMessagesOfPreviousGoal;
    }
}
```

**解决的问题**：目标 1 达成后切换到目标 2，LLM 判定目标 2 时只能看到 1-2 条消息，缺少上下文理解不了对话发展到了哪里。Bridge 补充了"上一个目标最后说了什么"，但明确标注"不作为判定依据"——避免 LLM 用前一目标的对话来判定当前目标。

#### 失败后的分支策略

```java
private void applyGoalJudgment(session, achieved, failed) {
    if (failed) {
        String flowMode = currentGoal.getPostFailureFlowMode();
        if (ScriptGoalFailureFlowType.END_DIALOGUE.equals(flowMode)) {
            // 策略1：直接终止对话（严格模式）
            session.setSessionStatus(TERMINATED);
            session.setEndReason(GOAL_FAILED);
        } else {
            // 策略2：跳过失败目标，继续下一关（宽松模式）
            session.setCurrentGoalIndex(currentIndex + 1);
        }
    }
}
```

每个目标可以独立配置失败后的处理策略：
- `END_DIALOGUE`：严格考试场景，一关失败就结束
- `CONTINUE_NEXT`：练习场景，失败也继续，重点是完整体验

#### 判定结果一致性保护

```java
// LLM 偶尔输出 achieved=true 且 failed=true（矛盾）
// 代码层做兜底保护
if (outcome.achieved && outcome.failed) {
    log.warn("contradictory judge result, treating as not achieved");
    outcome.achieved = false;
    outcome.failed = false;
}
```

#### 会话结束信号检测

Prompt 中明确区分了"结束"和"非结束"：
```
算结束：道别语（再见/拜拜/挂了/祝您生活愉快/感谢配合）
       严重违规

不算结束：
- 客户抱怨推脱但仍在对话
- 说"我想想"、"再说吧"
- 威胁投诉但继续陈述
- 学员简短回应"好的"
```

这些"不算结束"的反例是踩坑总结出来的——LLM 很容易把"我再想想吧"误判为结束信号。

### 2.4 目标判定与对话生成的时序设计

```
前端交互时序：
                                          
User → POST /sendMessage ───────────┐     
       (学员发消息)                  │     
                                    ▼     
                    [对话LLM生成回复]      
                    latency: 2-4s         
                                    │     
User ← 200 AI回复 ◄────────────────┘     
                                          
User → POST /judgeGoal ─────────────┐     
       (前端收到回复后立即调)         │     
                                    ▼     
                    [判定LLM评估目标]      
                    latency: 1-3s         
                                    │     
User ← 200 判定结果 ◄───────────────┘     
       (achieved/failed/transition)       
```

**为什么不把判定放在 sendMessage 内部？**
1. 减少用户等待：先返回回复（用户看得见），判定异步进行
2. 判定需要看到 AI 回复后的完整对话：AI 回复保存后再判，信息更全
3. 解耦：对话失败不影响判定，判定失败不影响对话

## 三、面试时如何讲这些难点

### 讲法 1：回答"你最复杂的技术挑战是什么"

> "对话质量调优。不是写个 Prompt 就完事——我们经历了 20+ 个版本的迭代。核心矛盾是：AI 要演得像真人（角色扮演），但又不能太强让学员说不了话。
>
> 举个具体例子：AI 扮演逾期客户，早期版本会无限拖延确认身份，反复质疑'你是谁？你怎么有我号码？'——10 轮都进不了业务环节。解决方案是在 Prompt 中加入量化硬约束：'最多1句反问，下一轮必须配合'。这比模糊的'保持适度防备'有效得多。
>
> 类似的坑有 6-7 个，每个都需要特定的 Prompt 设计模式来解决。这不是一次性写完的，是持续迭代的过程。"

### 讲法 2：回答"目标判定怎么做的"

> "最关键的架构决策是**对话生成和目标判定分离为两次独立 LLM 调用**。
>
> 早期版本让一个 LLM 同时'扮演角色'和'判断学员是否达标'，结果 AI 倾向于过早判定达成——因为它在角色扮演时有'把对话推向好的方向'的倾向，这就是 self-serving bias。
>
> 分离后，判定 LLM 是一个纯粹的裁判——它只看对话记录，按固定顺序（结束信号→违规→达成）逐条核对，不参与角色扮演。准确率从约 70% 提升到 90%+。"

### 讲法 3：回答"Prompt 怎么迭代的"

> "四个原则：
