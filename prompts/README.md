# Prompt 模板设计规范

## 一、模板约定

| 约定 | 规则 |
|------|------|
| 占位符 | `{{camelCaseVariable}}`，Mustache 风格 |
| 首行 | 必须包含 `{{responseLanguageInstruction}}`（i18n 切换） |
| 输出格式 | 模板末尾严格定义 JSON Schema + 示例 |
| Few-shot | 直接嵌入模板（3-6 个覆盖不同场景） |
| 文件命名 | `ai_{module}_{action}.md` |
| 注册 | 在 `CLASSPATH_BY_KEY` Map 中注册 key → 文件路径 |

## 二、模板结构标准

```markdown
# 角色
{{responseLanguageInstruction}}

你是XXX，负责YYY任务。

## 系统调用链路
（说明当前步骤在整体流程中的位置）

## 输入信息
（用 {{变量}} 注入的业务数据）

## 行为规则
（详细的行为约束，越具体越好）

## 输出格式
JSON: {"key": "type", ...}

## 示例
输入: ...
输出: {"key": "value", ...}
```

## 三、设计原则

### 1. 角色明确
每个 Prompt 开头明确定义 AI 的角色身份，避免泛泛而谈。

### 2. 链路感知
告诉 LLM 它在整个系统中的位置——上游是什么、下游消费它的输出干什么。这能帮助 LLM 理解输出格式的重要性。

### 3. 负面约束
不仅告诉 LLM "该做什么"，还要告诉它"不该做什么"。例如：
- "不要复述历史对话中已经明确说明的内容"
- "不要暴露你是 AI"
- "不要直接告诉学员标准答案"

### 4. 输出格式锚定
- 在模板末尾给出完整的 JSON 示例
- 字段名用英文（避免 LLM 输出中文 key）
- 说明每个字段的取值范围
- 对于数组，说明最大/最小长度

### 5. 分步指令
复杂任务拆成编号步骤（固定顺序），减少遗漏：
```
# 判定流程（固定顺序）
1. 判断结束信号 → sessionShouldEnd
2. 判断是否存在严重违规 → failed
3. 若未违规，判断达成条件 → achieved
```

### 6. 防幻觉机制
- 要求输出 evidence（引用原文作为依据）
- 限定输出长度（控制发散）
- 对于判定类任务，要求先 reasoning 再 conclusion

## 四、复用说明

以下 Prompt 模板可直接用于类似培训/对话/评分场景，只需：
1. 替换角色描述（催收 → 你的业务场景）
2. 调整行为规则
3. 修改输出字段

核心结构和设计原则是通用的。
